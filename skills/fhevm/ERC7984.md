# ERC7984.md — Confidential Token Standard

ERC-7984 is the confidential token standard for FHEVM. It extends ERC-20 with encrypted balances and transfer amounts. Balances and transfer values are `euint64` handles — nobody can see what anyone holds or sends.

---

## §1 — What ERC-7984 adds over ERC-20

| Feature | ERC-20 | ERC-7984 |
|---|---|---|
| Balances | Public `uint256` | Encrypted `euint64` |
| Transfer amounts | Public | Encrypted |
| Allowances | Public | Encrypted |
| Total supply | Public | Public (sum of encrypted balances) |
| Compliance access | None | `FHE.allow` to auditor |

The interface is intentionally familiar — `transfer`, `approve`, `transferFrom` work the same way. The difference is that amounts are `externalEuint64` inputs with proof, and balances are `euint64` handles.

---

## §2 — OpenZeppelin Confidential Contracts

OpenZeppelin ships a confidential contracts library for FHEVM. It includes:
- `ConfidentialERC20` — base ERC-7984 implementation
- `ConfidentialERC20Mintable` — with mint/burn
- `ConfidentialERC20Votes` — governance extension
- `ConfidentialERC20Wrapped` — wrap standard ERC-20 into confidential

Install:
```bash
npm install @openzeppelin/confidential-contracts --legacy-peer-deps
```

## §3 — Minimal ERC-7984 token

```solidity
// SPDX-License-Identifier: BSD-3-Clause-Clear
pragma solidity ^0.8.24;

import "@openzeppelin/confidential-contracts/token/ERC20/ConfidentialERC20.sol";

contract MyConfidentialToken is ConfidentialERC20 {
    constructor() ConfidentialERC20("My Token", "MTK") {}

    // Mint plaintext amount, stored as encrypted balance
    function mint(address to, uint64 amount) external {
        _mint(to, amount);
    }
}
```

That's it. `ConfidentialERC20` handles:
- Encrypted balance storage and updates
- `FHE.allowThis` + `FHE.allow` on all balance mutations
- Transfer validation in FHE (no negative balance possible)
- Standard ERC-20 interface compatibility

## §4 — Transferring confidential tokens

From a frontend:

```typescript
// Encrypt the transfer amount
const input = instance.createEncryptedInput(tokenAddress, signer.address);
input.add64(1000n); // 1000 tokens
const { handles, inputProof } = await input.encrypt();

// Transfer
const tx = await token.transfer(recipientAddress, handles[0], inputProof);
await tx.wait();
```

The recipient's balance increases by exactly 1000. Neither the sender, recipient, nor anyone else can see the amount — except via authorized user-decrypt.

## §5 — Wrapping ERC-20 into ERC-7984

Convert an existing public ERC-20 into a confidential version:

```solidity
import "@openzeppelin/confidential-contracts/token/ERC20/ConfidentialERC20Wrapped.sol";
import "@openzeppelin/zeppelin-solidity/contracts/token/ERC20/IERC20.sol";

contract WrappedUSDC is ConfidentialERC20Wrapped {
    constructor(address usdc) ConfidentialERC20Wrapped(IERC20(usdc), "Confidential USDC", "cUSDC") {}
}
```

Users deposit public USDC → receive encrypted cUSDC. Withdraw burns cUSDC → returns public USDC. This is exactly how Zama's own cUSDT works on mainnet.

## §6 — Reading encrypted balances

Only the balance owner (or explicitly granted addresses) can decrypt:

```typescript
// Get encrypted balance handle
const encBalance = await token.balanceOf(userAddress);
const handle = '0x' + BigInt(encBalance).toString(16).padStart(64, '0');

// User-decrypt (requires EIP-712 signature)
const { publicKey, privateKey } = instance.generateKeypair();
const startTimestamp = Math.floor(Date.now() / 1000);
const eip712 = instance.createEIP712(publicKey, [tokenAddress], startTimestamp, 1);
const { EIP712Domain: _, ...types } = eip712.types;
const sig = await signer.signTypedData(eip712.domain, types, eip712.message);

const result = await instance.userDecrypt(
  [{ handle, contractAddress: tokenAddress }],
  privateKey, publicKey, sig,
  [tokenAddress], userAddress, startTimestamp, 1
);

const balance = result[handle]; // plaintext BigInt
```

## §7 — Granting auditor access

```solidity
// In your contract — grant auditor read access to caller's balance
function grantAuditorAccess(address auditor) external {
    FHE.allow(encBalances[msg.sender], auditor);
}
```

The auditor can then call `userDecrypt` for the specific handle. No other balances are exposed.

## §8 — ERC-7984 vs rolling your own

**Use ERC-7984 / ConfidentialERC20 when:**
- You need a transferable token with private balances
- You want ERC-20 compatibility (DEX integration, wallets)
- You need wrapping/unwrapping of public tokens

**Roll your own `euint` storage when:**
- You need non-transferable encrypted values (splits, scores, votes)
- You need custom access control per-value per-address
- The asset is not a fungible token

RoyaltyLayer uses custom `euint16` storage (not ERC-7984) for splits because splits are non-transferable per-asset-per-holder values, not a fungible token. Revenue is paid in a `ConfidentialERC20` (ERC-7984) token.

## §9 — Common ERC-7984 mistakes

| Mistake | Fix |
|---|---|
| Calling `balanceOf` and passing raw `uint256` to SDK | Convert: `'0x' + BigInt(raw).toString(16).padStart(64,'0')` |
| Wrapping ERC-20 without approving first | Call `approve(wrappedAddress, amount)` on the underlying token before `wrap()` |
| Checking balance with `> 0` on-chain | Use `FHE.gt(balance, FHE.asEuint64(0))` — returns `ebool` |
| Emitting transfer amount in events | Never emit plaintext amounts — emit handles or nothing |
| Using `euint256` for balances | Use `euint64` — ERC-7984 specifies 64-bit. `euint256` only supports bitwise and eq. |
